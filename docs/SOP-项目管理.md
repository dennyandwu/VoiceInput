## 1. 需求收集与任务拆解

### 1.1 需求来源
- **0xFG** — 产品方向、功能需求、优先级决策
- **Satoshi (CTO)** — 技术需求、架构决策
- **Ansen (运维)** — 运维需求、基础设施
- **团队内部** — bug 修复、技术债务、优化建议

### 1.2 需求登记
所有需求统一登记到 **Obsidian Projects/** 目录：
- 编号规则: `{负责人}-{年份}-{序号}` (如 `SATOSHI-2026-001`)
- 每个项目一个文件夹，包含阶段文档

### 1.3 任务拆解
1. 收到需求后 24h 内完成技术调研
2. 拆解为 Phase（P1/P2/P3...），每个 Phase ≤ 3 天
3. 明确每个 Phase 的：交付物、验收标准、依赖项
4. 在项目总表（`开发中心项目总表.md`）登记排期

### 1.4 角色分工

| 角色 | 负责人 | 职责 |
|------|--------|------|
| **产品决策** | 0xFG | 需求确认、优先级、最终验收 |
| **项目管理** | Issac | 排期、进度跟踪、风险管理、SOP 维护 |
| **技术开发** | Satoshi | 架构设计、编码、代码质量、技术文档 |

## 2. 开发迭代流程

### 2.1 版本号规范 (SemVer)
```
v{MAJOR}.{MINOR}.{PATCH}
```
- **MAJOR** — 不兼容的重大变更
- **MINOR** — 新功能（向后兼容）
- **PATCH** — bug 修复、小优化

示例: `v0.3.0` → `v0.3.1`(bugfix) → `v0.4.0`(新功能) → `v1.0.0`(正式发布)

### 2.2 Git 分支策略

```
main          ← 稳定发布分支，只接受 PR 合入
├── dev       ← 开发主分支，日常开发在此
├── feature/* ← 功能分支 (feature/vad-improvement)
├── bugfix/*  ← 修复分支 (bugfix/hotkey-crash)
└── hotfix/*  ← 紧急修复 (hotfix/v0.3.1)
```

**分支规则:**
- `main` 只通过 PR 合入，禁止直接 push
- `dev` 为日常开发分支，feature 分支从 dev 拉出
- 功能完成后 PR → dev，经 review 后合入
- 发版时 dev → main，打 tag

### 2.3 迭代节奏
- **Sprint**: 1 周为一个迭代
- **每日 09:00**: 各 agent 通过 A2A 汇报进展
- **每日 09:30**: Issac 汇总更新项目总表
- **每周五**: Sprint 回顾，下周排期

## 3. 进度跟踪与汇报

### 3.1 每日站会 (Daily Standup)
- **时间**: 09:00 (cron 触发 A2A 催报)
- **格式**:
```
[项目编号] 
昨日完成: xxx
今日计划: xxx
阻塞项: xxx (无则写"无")
```
- **汇总**: Issac 09:30 更新 `开发中心项目总表.md`

### 3.2 项目状态标记

| 标记 | 含义 |
|------|------|
| 🟢 正常 | 按计划推进 |
| 🟡 进行中 | 有小延迟但可控 |
| 🔴 阻塞 | 需要介入解决 |
| ✅ 已完成 | 已交付验收 |
| ⏸️ 暂停 | 等待外部依赖 |

### 3.3 升级机制
- 阻塞 **1 天** → Issac 主动协调
- 阻塞 **2 天** → 升级到 0xFG 决策
- 阻塞 **3 天+** → P0 红色警报，重新评估方案

## 4. 风险管理

### 4.1 风险识别
每个 Phase 启动时识别风险，记录在项目文档中：
- **技术风险**: 依赖库兼容性、性能瓶颈
- **资源风险**: agent 并发任务、Mac mini 资源
- **外部风险**: API 变更、第三方服务不可用

### 4.2 风险等级

| 等级 | 影响 | 应对 |
|------|------|------|
| **高** | 项目可能失败 | 立即上报 0xFG，准备 Plan B |
| **中** | 延期 1-3 天 | Issac 协调资源 |
| **低** | 可忽略 | 记录观察 |

### 4.3 升级路径
```
开发者 → Issac (PMO) → 0xFG (决策)
                ↓
           Ansen (运维支持)
```

## 5. 项目报告

### 5.1 报告存放
- 路径: `Obsidian Vault/Projects/{项目编号}/`
- 编号: `ISSAC-{年份}-{序号}`

### 5.2 报告内容
每个项目或系统更新须包含：
1. **背景** — 为什么做
2. **方案** — 怎么做
3. **实施** — 做了什么
4. **验收** — 怎么验证
5. **风险** — 潜在问题
6. **总结** — 经验教训
